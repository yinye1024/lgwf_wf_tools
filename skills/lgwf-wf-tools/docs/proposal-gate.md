# 模糊修改 Proposal Gate

本文用于 `ambiguous_modify_goal` 场景：用户目标可能需要改文件或启动 workflow，但范围和授权不清。

## 触发条件

满足以下情况时，先进入模糊请求 proposal gate：

- 用户说“修复优化”“优化修复”“完善”“处理”“整理”“质量提升”等范围词。
- 用户提出模糊的“优化/修复”。
- 请求同时给出目标目录、目标 workflow 或 package 路径。
- 请求没有明确授权“直接修改”“立即运行”“自动执行”。

## 处理规则

- 除非用户明确说“直接修改”或给出等价授权，否则先输出 proposal。
- proposal 前禁止改文件，禁止启动任何内部 workflow，禁止启动目标 workflow run。
- 只做只读检查、证据整理和候选路由判断。

## Proposal 内容

proposal 必须包含：

- 目标
- 发现依据
- 候选路由
- 修改范围
- 是否真实运行
- approval 边界
- 最小验证方式
- 明确的 will-not-do

行文中必须明确覆盖“目标、发现依据、候选路由、修改范围”。
